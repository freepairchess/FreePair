# FreePair v0.5.20260604

Same-day respin of v0.4.20260604 with one fix to the in-app **Current version** label.

## What's fixed

The Settings → Auto-update **Current version** row was showing `FreePair v1.0.0+<commit-sha>` instead of the actual release version. The release script was calling `dotnet publish` without any version MSBuild properties, so the published assembly kept the project default (`1.0.0`) and the .NET SDK auto-appended the commit SHA via SourceLink. The new "Current version" label reads from that assembly metadata, so it dutifully displayed what was actually in the binary — just not the version the TD installed.

`release\release.ps1` now stamps the publish command with `/p:Version=…` and `/p:InformationalVersion=…`, and disables the SDK's commit-SHA suffix. The label now correctly reads `FreePair v0.5.20260604`, matching the GitHub release tag and the installer filename. File Explorer → Properties → Details shows the same.

No other code changes. Same USCF pairing engine, same Settings UI, same installer payload as v0.4.20260604.

## Installer

Self-contained Windows x64 installer (`FreePair-win-x64-Setup.exe`) and a no-install portable build (`FreePair-win-x64-Portable.zip`). Both bundle:

- the FreePair desktop application (Avalonia UI),
- the FreePair USCF pairing engine,
- the FIDE Dutch pairing engine (BBP Pairings),
- the .NET 10 runtime — **no separate runtime install required**.

The auto-updater (Velopack) is included. Settings → Auto-update now correctly shows your installed version and gives you a "Check now" button for on-demand update checks.

## Known limitations

Unchanged from v0.4.20260604 / v0.3.20260604. See those release notes and `docs/USCF_DISCRETIONARY_DIVERGENCES.md`.
