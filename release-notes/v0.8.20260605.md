# FreePair v0.8.20260605

FreePair is a free, open-source desktop application for chess-tournament directors. It imports SwissSys 11 `.sjson` files, runs USCF Swiss and FIDE Dutch pairings (via FreePair's own USCF engine and the BBP Pairings FIDE engine, both bundled), and gives the TD end-to-end tools to pair, score, intervene, and publish reports.

This release is a transparency and correctness pass on the USCF rules surface: the in-app explanations are now precise, every USCF rule citation is clickable, and the engine quietly enforces the USCF rule book's rating-difference caps on colour swaps.

## What's new in v0.8

- **Click any USCF rule citation in the "Why this pairing?" dialog to read the rule.** Every annotation line (transposition, interchange, colour decision, floater drop, bye assignment, fallback) ends with the rule that drove the decision — e.g. `(USCF 29C2 — Other adjustments)` or `(USCF 29E5f — Colors in a series — no three in a row)`. The whole parenthetical is now a clickable hyperlink that opens that rule's section in the FreePair USCF rules reference in your default browser. No more flipping to a separate document to find what `29E5f` means — the title is right there, and one click gets you the full text and worked example.
- **Plain-language rule titles inline.** The bare rule numbers from earlier releases are gone. Citations now read like `(USCF 29C2 — Other adjustments)` so even without clicking, the TD knows what the cited rule is about.
- **Every USCF citation in the engine is now correct.** A systematic audit found that the previous "Why this pairing?" dialog cited several wrong or non-existent rule numbers (for example: it called transpositions "USCF 28L1", but the real 28L1 is "Explanation and display" for byes; it cited "29D5" for an absolute colour streak, but rule 29D5 doesn't exist — the right authority is 29E5f). All 13 citations the engine emits are now reconciled with the USCF *Official Rules of Chess* (7th edition).
- **Defensive enforcement of the USCF 80-point and 200-point rules on colour swaps.** When the engine considers swapping two players in a score group to reduce a colour conflict, it now checks the rating-difference caps the rule book imposes: alternation-only conflicts (mild claims) require ≤80 points (29E5a), equalisation conflicts (strong claims) allow up to 200 points (29E5b), and either-player-is-unrated swaps are exempt (29E5g). The transposition cost is evaluated correctly per 29E5c as the smaller of the two rating-difference views. This caps engine behaviour the rule book always required but the previous reducer didn't check; no measurable change to SwissSys-fidelity numbers on the current test corpus (54 of 81 sections continue to match SwissSys exactly), but the engine no longer makes swaps the rule book would reject as overshooting.
- **`docs/USCF_RULES.md` is now bundled in the public release.** The 100 KB authoritative rules reference is committed alongside each release, so the in-app dialog's hyperlinks resolve even years after the release tag, regardless of what happens to the dev source. The doc covers chapters 22, 27, 28, and 29 — roughly 95 rules — with plain-language statements, worked examples for the tricky 29E5 family, and an honest FreePair-coverage label on every rule (enforced, partial, planned, TD discretion, or deferred).

## Known limitations (still planned for later releases)

- **Engine still has 27 known SwissSys divergences** on the curated regression corpus, all catalogued as future-work branches in `docs/USCF_RULES_COVERAGE.md`. The most impactful (per coverage analysis): hard "no three in a row" cap (29E5f), colour-driven interchanges across the upper/lower half boundary (29E5d), and the half-group-due Look Ahead heuristic (29E6a). None of these affect the headline transparency / correctness work in v0.8.
- **`28L3` forfeit-win-bye exclusion** and **`28L4` "unless all others have already had a bye" escape** are partial — small bye-picker refinements scoped as their own future branches.
- **Accelerated pairings (28R), reentries (28S), class pairings (29I)** remain explicitly deferred. TDs needing these workflows continue to use SwissSys for the affected portion of the event.
