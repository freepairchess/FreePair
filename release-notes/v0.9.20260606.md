# FreePair v0.9.20260606

FreePair is a free, open-source desktop application for chess-tournament directors. It imports SwissSys 11 `.sjson` files, runs USCF Swiss and FIDE Dutch pairings (via FreePair's own USCF engine and the BBP Pairings FIDE engine, both bundled), and gives the TD end-to-end tools to pair, score, intervene, and publish reports.

This is a **stability and internal-hardening release**. It carries no change to the pairings or colours your events produce — every section pairs exactly as it did in v0.8 — but it removes a rare way the pairing engine could stall on large late-round sections, and it lays in (switched off) the groundwork for a future colour-balancing improvement. If v0.8 is working well for you, upgrading is low-risk and recommended.

## What's new in v0.9

- **The pairing engine can no longer stall on large, late-round sections.** The colour-balancing search that runs while pairing a round is now bounded by a single, shared work budget that applies across the whole of a score group's optimisation — not per-attempt. A specific combination (a big section, late in the event, with several players needing the same colour) could previously make that search do far more work than necessary; on the largest test sections it could appear to hang. This release caps the total work so pairing always returns promptly, with no change to the pairings produced on any normal event. Pure robustness — you should simply never see a slow pair-a-round again.
- **Groundwork for cross-half colour interchanges (USCF 29E5d), shipped switched off.** The engine now contains a complete second-pass colour optimiser that can swap a player across the upper/lower-half boundary to clear a colour clash the usual same-half swaps can't — and the "Why this pairing?" dialog is ready to show it with a clickable **29E5d** rule citation. It is **disabled by default** in this release: enabling it today would change a handful of sections away from how SwissSys pairs them, and we don't ship a divergence from SwissSys without a faithful rule for *when* the rule book actually wants the interchange. The machinery, the dialog citation, and its tests all ship now so the feature can be turned on cleanly the moment that rule is pinned down.
- **Honest rule-coverage labelling.** The bundled USCF rules reference and coverage board now describe rule **29E5d** as *partial — implemented but gated off*, with a short note on exactly why and what would turn it on. No more "planned" placeholder for work that is in fact written and waiting.

## Known limitations (still planned for later releases)

- **SwissSys fidelity is unchanged from v0.8.** The engine still matches the curated regression corpus on the same set of sections it did in v0.8; this release deliberately holds fidelity flat while hardening robustness. The known divergences remain catalogued as future-work branches in `docs/USCF_RULES_COVERAGE.md`.
- **Cross-half colour interchange (29E5d) is present but switched off.** It will be enabled in a future release once there is a validated rule for when SwissSys takes an interchange versus tolerating the colour clash — turning it on prematurely would trade SwissSys agreement on several sections for a small gain on a couple of others.
- **`28L3` forfeit-win-bye exclusion** and **`28L4` "unless all others have already had a bye" escape** remain partial — small bye-picker refinements scoped as their own future branches.
